Introduction

This documentation outlines key observations, known limitations, and conventions related to the GeoNames data dumps and ontologies used in the GeoNames SPARQL interface.

1. Use of dcterms:modified vs. Absence of dcterms:created in CSV Dumps

The dcterms:created property is not included in the GeoNames CSV dump files. Only dcterms:modified is present to indicate the last update date.

2. Namespace Prefix gn Variations Across RDF Files

  • The gn prefix in GeoNames RDF files is not consistent in protocol:
    • The ontology file (ontology_v3.3.rdf) uses https://www.geonames.org/ontology#.
    • The mappings file (mappings_v3.01.rdf) and place descriptions (about.rdf) use http://www.geonames.org/ontology#.
  • To ensure security and consistency, this project adopts the https://www.geonames.org/ontology# namespace as specified by the ontology file.

3. Inconsistencies in gn:alternateName Usage

The gn:alternateName property usage varies across GeoNames RDF datasets:

  • Some alternate names lack language tags.
  • Language tags vary reflecting source dataset differences (e.g., de, en).
  • These inconsistencies are inherited from the original data sources.

4. Handling Population Values Set to Zero

Population data in GeoNames has these characteristics:

  • Many CSV dumps list population as 0 when data is missing, not truly zero population.
  • The about.rdf files omit the population property if unknown.
  • Our Turtle-format output excludes gn:population triples with a value of zero to prevent misunderstanding.

Example: München (Uebigau-Wahrenbrück) shows a population of 0 in CSV, no value in RDF, despite external sources reporting an estimated 16 residents.
